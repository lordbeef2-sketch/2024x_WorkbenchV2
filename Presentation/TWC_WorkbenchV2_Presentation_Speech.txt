TWC Workbench V2 — By Raymond Reeves

Slide 1 — Title
This presentation explains the updated TWC Workbench V2 at a practical architecture level. The main point is that Workbench now carries a complete Cameo model tree into a secure web workspace, a scoped Developer API, and a grounded Agent workflow. Teamwork Cloud access, authentication, tokens, cache materialization, plugin ingestion, and knowledge push remain controlled by the FastAPI backend.

Slide 2 — What the app is built to solve
The problem this app solves is controlled access to Teamwork Cloud data through a web interface. The frontend gives the user the experience: dashboards, project browsing, model browsing, details, compare, developer API, and admin tools. The backend keeps the sensitive parts server-side: authentication, sessions, tokens, capability checks, and direct Teamwork Cloud calls. The third major purpose is speed and automation: Workbench can use cache-backed data and ingest snapshots or deltas from the Cameo plugin.

Slide 3 — High-level architecture
At a high level, the browser only talks to the Workbench backend. The backend is the control point. It serves the frontend, manages secure sessions, routes requests through auth, server, workspace, cache, and Agent APIs, then talks outward to Teamwork Cloud and the Authentication Server. The Cameo plugin now walks every loaded model root, including attached modules, and publishes ordered snapshots or deltas into the cache ingest API. That same normalized branch data feeds the browser, Developer API, and knowledge push.

Slide 4 — User sign-in and server selection flow
The normal sign-in flow starts on the landing page with preset server selection. When the user clicks Sign In via TWC, the backend builds a server-specific authorization URL and redirects the browser to the TWC Authentication Server. After login, the browser returns to the Workbench callback. The backend exchanges the code for tokens and validates the user against Teamwork Cloud. The result is a secure Workbench session. There is also an explicit token fallback for environments where direct AuthServer flow needs troubleshooting.

Slide 5 — Frontend experience: workspace tabs
The frontend is a React and TypeScript application using Material UI and TanStack Query. The Model Browser requests the complete accessible containment tree up front and preserves Cameo ownership order, so search and filtering see every loaded node. Item Details now separates All Cameo Properties from Stereotypes and Tags, including unset defaults, calculated or derived values, type and multiplicity metadata, and structured references. Developer API provides scoped keys plus runnable full-tree, direct-children, and native-specification Python examples. The Agent is again a visible top-level workspace, with persistent 3DS reference status, branch knowledge status, push controls, and grounded model chat.

Slide 6 — Backend responsibilities
The backend is where the application actually becomes enterprise-safe. The route groups expose clean areas of functionality: authentication, server management, workspace operations, and cache operations. Behind the routes, PlatformService is the orchestration layer. It decides how to use sessions, server profiles, adapters, cache, jobs, API keys, and agent settings. The Teamwork adapters isolate TWC version differences. Storage is handled through a SQLite repository by default, which keeps deployment simple but still gives the app durable state.

Slide 7 — Verified Teamwork Cloud contract boundary
A major design choice is that the app is contract-bound. It uses the verified Swagger file instead of inventing endpoints. That matters because Teamwork Cloud surfaces can vary by version and deployment. The app uses the verified parts: workspaces, resources, branches, models, elements, revision diff, and current-user validation. If an area is not defined in the Swagger contract, the app does not pretend it is safely supported. Version-specific behavior is handled behind adapter and capability layers.

Slide 8 — Native Cameo specifications and Developer API
The cache design stores each shared branch payload once while keeping user permission overlays separate. The tree endpoint returns the complete accessible containment tree when depth is omitted, while the children endpoint remains available for large-tree integrations. Specification schema version 2.0 exposes every native metamodel feature together with set state, defaults, derived and read-only metadata, type, bounds, ordering, uniqueness, and structured values. It also exposes each applied stereotype and its ordered inherited or calculated property values. The Developer API carries that schema through scoped read, write, and edit keys, and the UI and repository include runnable Python examples.

Slide 9 — Cameo plugin ingestion path
The Cameo plugin is the desktop-to-web bridge. It enumerates all project models instead of stopping at the primary model, which brings attached-module roots into the published snapshot. For each element it exports every EStructuralFeature, including unset defaults and derived or read-only metadata, plus structured references, diagrams, and ordered inherited or calculated stereotype values through the official Cameo helper APIs. On save or close, it publishes a delta when a baseline exists, or a full snapshot when needed. This is an authoritative headless representation of Cameo model and stereotype content; it does not scrape Swing controls from the Specification dialog.

Slide 10 — Security and deployment model
Security is mostly about where control lives. Workbench keeps tokens on the backend, not in browser local storage. It uses server-side session management and secure HTTP-only cookies. Server profiles are centrally configured, so users pick from approved Teamwork Cloud targets instead of manually entering random servers. The launcher can run it as a single-origin application where FastAPI serves both the API and the built React frontend, which helps avoid cross-origin deployment problems.

Slide 11 — End-to-end data flow
The normal request flow is straightforward. A user does something in the React workspace. The frontend API service sends a request to FastAPI. The route validates the session and delegates to PlatformService. PlatformService then decides whether the request should go live to Teamwork Cloud through an adapter, use the Workbench cache, run a job, modify storage, or return configuration state. The response is normalized and returned to React Query, which updates the visible workspace.

Slide 12 — Validated structure, live-runtime proof still required
The backend modules compile, the frontend type-checks and builds, and the configured 2024x 3DS knowledge base resolves 456 pages and 857 chunks. The exporter follows the official 2024x metamodel and TagsHelper signatures for native properties and applied stereotype values. The remaining proof belongs in the target environment: authentication, TLS, redirects, live Teamwork Cloud behavior, compilation and capture with the installed Cameo SDK, and an actual Open WebUI upload and chat must still be validated end to end.

Slide 13 — Bottom line
The bottom line is that TWC Workbench V2 now carries the complete accessible Cameo tree and native specification content into every useful surface. Users get faithful navigation and element detail. Developers get scoped APIs and runnable examples. Every mapped Open WebUI model receives two processed sources on each Agent turn: a persistent, fingerprinted Workbench and 3DS operating reference, plus the permission-scoped selected branch model. The system prompt directs the model to retrieve the persistent reference for Workbench, Cameo, 3DS, and API questions and the branch source for project facts, without inventing either.
